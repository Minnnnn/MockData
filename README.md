# MockData

基于 Next.js 16 的 OpenAPI Mock 工作台。

这个项目提供一套从 OpenAPI JSON 导入、接口筛选、TS 类型生成、Mock 数据生成、本地 Mock 服务启动，到 AI 调优、热更新与本地缓存复用的完整闭环，适合前后端联调、接口演示和快速造数。

## 功能概览

- 上传并解析 OpenAPI JSON
- 按 `tag` 分组选择需要参与生成的接口
- 生成并下载 `types.ts` 与 `api.ts`
- 生成结构化 Mock 数据并进入工作区管理
- 启动本地 Mock 服务，支持 `GET/POST/PUT/PATCH/DELETE/OPTIONS/HEAD`
- 单接口调整状态码、延迟和返回内容，并支持热更新
- 支持批量 AI 调优与单接口内联 AI 调优
- AI 调优结果与手动热更新结果持久化到 IndexedDB
- 刷新 Mock 数据时按当前接口清理历史缓存并重新生成
- 工作区预览区展示“最终实际返回值”，与真实 `/mock-api` 响应保持一致

## 当前工作流

### 1. 上传并解析 OpenAPI

首页支持上传或直接粘贴 OpenAPI JSON。

支持能力：

- 解析接口、响应结构、请求结构、tag、状态码
- 统计接口数量
- 按 `tag` 分组展示
- 默认全选解析出的接口
- 将工作区基础数据写入 `sessionStorage`

### 2. 进入工作区

工作区分为两个 Tab：

- `TS 类型生成`
- `Mock 数据（3步）`

并提供 `重置工作区` 按钮，用于清空当前工作区状态并回到上传页。

### 3. TS 类型生成

支持：

- 基于当前勾选接口生成 `types.ts`
- 生成 `api.ts`
- 在线预览
- 下载 `types.ts`
- 下载 `api.ts`
- 在接口勾选变化后重新生成

当前实现使用：

- `swagger-typescript-api`

### 4. Mock 数据流程

Mock 数据流程固定为 3 步：

1. 接口确认
2. 策略生成
3. 服务状态

#### Step 1：接口确认

支持：

- 按 tag 分组查看接口
- 卡片式勾选接口
- `全选`
- `反选`

#### Step 2：策略生成

当前保留的策略项：

- 数据条数

说明：

- 仅对分页接口或 `data` 为数组的接口生效
- 普通对象接口固定生成 1 条

#### Step 3：服务状态

每个接口支持：

- 设置状态码
- 设置延迟毫秒数
- 查看请求调用示例 `curl`
- 查看并编辑“实际返回预览”
- `热更新`
- `AI 调优`
- 查看请求参数提示
- 查看接口更新状态：默认、loading、success、error

额外支持：

- `刷新 Mock 数据`
  会清空当前已选接口在 IndexedDB 中的历史调优缓存，重新生成 Mock，并自动热更新到服务
- `重新启动服务`

## AI 调优

项目已接入 AI 调优能力，既支持批量调优，也支持单接口内联调优。

当前能力包括：

- 右下角对话面板支持批量选择多个接口统一调优
- 单接口卡片内支持内联输入调优要求
- AI 调优后结果立即写回工作区
- AI 调优结果持久化到 IndexedDB，后续优先复用
- AI 调优成功后，预览区回显“最终实际返回值”
- 若模型执行成功但内容未变化，界面会明确提示“没有可回填变更”
- 服务端会打印 AI 调用日志，包含请求摘要、模型原始响应摘要和变更字段

当前 AI 接口实现位于：

- `app/api/mock-workflow/route.ts`

当前 AI 调用方式：

- SDK：`openai`
- Provider：由 `BASE_URL` 指定，当前默认配置为 DeepSeek
- Model：`deepseek-chat`
- Key 来源：`API_KEY`

## Mock 服务能力

本地 Mock 服务通过项目内 API 路由完成配置与启动。

当前支持：

- 服务启动
- 服务重置
- 路由动态配置
- 单接口状态码与延迟热更新
- 实际接口返回动态读取当前配置

Mock 服务相关接口：

- `app/api/mock-server/route.ts`
- `app/api/mock/[...slug]/route.ts`

默认访问前缀：

- `/mock-api`

例如：

```bash
curl http://localhost:3666/mock-api/your/path
```

## 实际返回预览与热更新

服务状态页中的预览区现在展示的是“最终实际返回值”，而不是内部原始 mock payload。

当前行为：

- 预览区内容与实际 `/mock-api/...` 响应语义一致
- 可直接编辑最终响应 JSON
- 点击 `热更新` 后，会将最终响应体反解为内部 payload，并同步更新服务和 IndexedDB
- 热更新成功后：
  - 当前接口标题显示成功状态
  - 当前接口调优数据写入 IndexedDB
  - 实际接口返回立即更新
- 热更新失败后：
  - 当前接口标题显示失败状态
  - 页面显示明确错误提示

### 预览编辑规则

- 当状态码为 `200` 时，预览区应编辑 `{ rc, msg, data }` 结构
- 热更新时仅将 `data` 反解回内部 payload，避免把 `rc/msg` 当业务字段存入 mock
- 当状态码为非 `200` 时，预览区按错误响应结构编辑

## 接口返回规则

当前接口返回封装规则：

- 成功响应默认返回：

```json
{
  "rc": 0,
  "msg": "",
  "data": {}
}
```

- 当接口状态码为 `403` 时，默认返回：

```json
{
  "rc": 1,
  "msg": "请登录",
  "data": null
}
```

- 当接口状态码为其他非 `200` 值时，默认返回：

```json
{
  "rc": 1,
  "msg": "接口错误",
  "data": null
}
```

- 如果原始 mock payload 本身已经是 `{ rc, msg, data }` 结构，成功响应阶段会优先取其中的 `data` 再重新封装，保证最终返回结构统一

## 分页接口规则

当前分页处理按接口语义区分：

### 普通数组接口

如果返回结构是：

```json
{
  "rc": 0,
  "msg": "",
  "data": []
}
```

规则：

- `page/pageSize` 不影响 `data` 数组数量
- 始终按当前配置的完整数组返回

### 分页对象接口

如果返回结构是：

```json
{
  "rc": 0,
  "msg": "",
  "data": {
    "items": [],
    "total": 0,
    "hasMore": false
  }
}
```

或：

```json
{
  "rc": 0,
  "msg": "",
  "data": {
    "item": [],
    "total": 0,
    "hasMore": false
  }
}
```

规则：

- `page/pageSize` 只影响 `data.items` 或 `data.item` 的返回数量
- `total` 取当前配置的总数据量
- 不传分页参数时，返回完整内容

## 数据持久化

当前项目同时使用两类本地存储。

### SessionStorage

用于保存当前工作区基础状态：

- OpenAPI 文本
- 解析结果
- 已选择接口
- 生成后的 TS 内容

### IndexedDB

用于保存 AI 调优后或手动热更新后的接口数据。

特性：

- 以 `workspaceId + endpointId` 为 key
- 同一份 OpenAPI 文档稳定命中同一个工作区 ID
- 服务启动时优先读取已调优数据
- 刷新 Mock 数据时可按当前已选接口维度清理缓存

相关实现位于：

- `lib/tuned-mock-db.ts`

## 挑战部分：问题与场景

### 你在工作中遇到了什么困难或挑战？

在前后端联调、测试造数、接口演示这类日常研发场景里，最常见的问题不是“接口没有定义”，而是“接口定义有了，但很难快速得到可联调、可修改、可复用的高质量 Mock 数据”。

具体业务场景：

- 前端开发人员在页面开发阶段，需要尽快接入接口并完成页面联调
- 后端接口尚未完成，或接口已完成但返回数据不稳定
- 测试人员需要覆盖正常、异常、分页、空数据、图片字段等多种返回情况
- 产品或业务同学需要在演示环境中看到更接近真实业务的数据效果

这类问题的典型表现：

- OpenAPI 文档已经存在，但接口多、字段复杂，手工造数据效率很低
- 临时写死 JSON 或本地 mock 文件容易和文档脱节
- 单次改完数据后无法沉淀，下次还要重新改
- 图片、分页、状态码、延迟、异常态等场景覆盖不完整
- 前端想改某个接口返回值时，往往还要找后端或重新改本地代码

本质上，这是“接口定义到可用 Mock 服务”的最后一公里效率问题。

### 过去你是怎么处理的？

过去的传统处理方式通常有几种：

- 手写本地 JSON 文件
- 在代码里临时写死 mock 数据
- 让前端自己维护一套简单 mock
- 等后端接口完成后再联调
- 用通用 mock 工具生成一版数据后，再手工逐个改字段

这些方式的主要痛点：

- 手工成本高
- 接口一多就难维护
- 改完后无法复用
- 数据不够真实
- 多人协作时容易出现“你本地能跑，我这里不一致”
- 一旦接口字段变化，TS 类型、Mock 数据、联调服务要分别更新

### 你希望解决后达到什么效果？（请量化）

希望实现的目标是：

- 单份 OpenAPI 文档从导入到启动可联调 Mock 服务，控制在 3 分钟内
- 单接口返回数据修改时间从 5 到 10 分钟降到 30 秒内
- 接口造数与联调准备时间减少 70% 以上
- 常见联调错误率降低 50% 以上
- 分页、异常态、图片字段等高频场景覆盖率提升到 90% 以上
- AI 调优后可复用，减少重复造数工作 60% 以上

## 解决方案

### 你是如何解决的？（解决方案中应当包含 AI 技术的应用）

我将问题拆成五个连续环节，并把它们整合为一个统一的工作台：

1. 解析 OpenAPI JSON
2. 按接口选择生成 TS 类型
3. 基于 schema 自动生成 Mock 数据
4. 启动本地 Mock 服务并支持热更新
5. 用 AI 对返回数据进行对话式调优，并持久化结果

核心解决思路不是单纯“生成一份 mock”，而是让 OpenAPI 文档直接成为工作区入口，把 TS、Mock、服务、AI 调优、缓存全部串成闭环。

AI 技术的应用方式：

- 在服务状态页和右下角对话面板中接入 AI 调优能力
- 将当前接口已有的 mock JSON 作为上下文
- 通过 `OpenAI SDK + DeepSeek` 调用 `deepseek-chat`
- 让模型只返回合法 JSON
- 将 AI 返回结果直接回填到对应接口的 Mock 数据区
- 同时把调优后的结果写入 IndexedDB，后续优先复用

这样做的价值是：

- 用户不需要改代码，只需要描述“把这个接口改得更像真实电商订单数据”
- 调优后的结果不会丢失
- 服务返回和界面预览始终保持一致

### 关键步骤说明

#### 1. OpenAPI 解析与接口筛选

- 上传 OpenAPI JSON
- 解析接口、请求结构、响应结构、tag、状态码
- 按 tag 分组展示
- 勾选本次需要参与 TS 与 Mock 生成的接口

#### 2. TS 类型生成

- 基于已勾选接口裁剪 OpenAPI 文档
- 使用 `swagger-typescript-api` 生成 `types.ts` 与 `api.ts`
- 支持页面预览和直接下载

#### 3. Mock 数据生成

- 基于响应 schema 递归生成结构化数据
- 针对邮箱、手机号、时间、金额、图片等字段做规则推断
- 对分页接口和数组接口支持批量生成
- 对 `rc` 字段做统一约束，默认固定为 `0`

#### 4. 本地 Mock 服务

- 将当前所有选中接口配置成可访问路由
- 支持状态码、延迟、返回内容热更新
- 支持请求方式匹配与分页数据切片
- 支持图片字段和图片数组字段特殊处理

#### 5. AI 调优与缓存复用

- 在右下角提供对话式 AI 调优入口
- 支持批量选择多个接口统一调整
- 单接口也可以在卡片中直接 AI 调优
- AI 返回结果即时回填
- 写入 IndexedDB，后续服务启动时优先使用历史调优结果
- 点击“刷新 Mock 数据”时，再清理当前接口缓存并重生成

## 新问题与解决

### 在使用新办法的过程中，有遇到新问题吗？

有，主要出现了三类新问题：

1. AI 调优后的数据虽然变了，但服务实际返回没有同步更新
2. 多次生成 Mock 后，历史调优数据和新生成数据之间会互相覆盖
3. 图片字段和图片数组字段在真实业务场景下不够稳定，展示效果不统一

### 如果有，是如何解决的？

针对这些问题，分别做了处理：

- 对“热更新”链路补齐成完整闭环：
  编辑 JSON -> 校验 -> 重配路由 -> 写入 IndexedDB -> 成功/失败反馈
- 为工作区引入 `workspaceId + endpointId` 维度的 IndexedDB 存储：
  让历史调优数据可以稳定命中，也可以按当前接口精确清理
- 增加“刷新 Mock 数据”机制：
  点击后先删除当前选择接口的 IndexedDB 缓存，再重新生成并自动热更新
- 为图片字段增加统一规则：
  从 `FIXED_IMAGE_URL` 中随机取值，数组字段则随机返回任意数量子集
- 为每个接口卡片增加操作状态反馈：
  热更新中显示 Spinner，成功显示 success，失败显示 danger

## 适用场景推广

### 在你看来，其他人在什么场景下也可以使用这个方法？

这套方法不仅适用于当前项目，也适合下面这些场景：

- 前端团队需要在后端接口未完成前独立推进开发
- 测试团队需要快速构造多种接口返回场景

- 多人协作项目中，希望统一 Mock 来源和调优结果
- 需要基于 OpenAPI 文档快速生成一套“可联调 + 可调整 + 可复用”的本地接口环境

可推广的核心不在某一个具体业务，而在于：

- 只要有 OpenAPI 文档
- 只要需要本地联调

这套方法就可以直接复用或稍作改造后推广使用。
