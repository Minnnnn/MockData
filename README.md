# MockData

基于 Next.js 16 的 OpenAPI Mock 工作台。

这个项目提供一套从 OpenAPI JSON 导入、接口筛选、TS 类型生成、Mock 数据生成、本地 Mock 服务启动，到 AI 对话调优与热更新的完整闭环。当前界面和能力已经围绕“工作区”模式整理完成，适合前后端联调、接口演示和快速造数。

## 功能概览

- OpenAPI JSON 上传与解析
- 按 `tag` 分组选择需要参与生成的接口
- 生成并下载 `types.ts` 与 `api.ts`
- 生成结构化 Mock 数据，并进入工作区管理
- 启动本地 Mock 服务，支持 `GET/POST/PUT/PATCH/DELETE/OPTIONS/HEAD`
- 单接口修改状态码、延迟、返回数据，并支持热更新
- 支持 AI 对话批量调优接口数据
- 支持单接口内联 AI 调优输入面板
- AI 调优结果持久化到 IndexedDB，后续优先复用
- 支持刷新 Mock 数据时清空当前接口的 IndexedDB 历史调优缓存
- 图片字段自动从 `FIXED_IMAGE_URL` 中随机返回

## 当前工作流

### 1. 上传并解析 OpenAPI

首页支持上传 OpenAPI JSON，并展示解析后的接口列表。

支持能力：

- 接口数量统计
- 按 `tag` 分组展示
- 勾选要参与后续 TS 生成和 Mock 生成的接口
- 解析成功后将工作区基础数据写入 `sessionStorage`

### 2. 进入工作区

工作区分为两个 Tab：

- `TS 类型生成`
- `Mock 数据（3步）`

并提供 `重置工作区` 按钮，可清空当前工作区状态并回到上传页。

### 3. TS 类型生成

支持：

- 基于当前勾选接口生成 `types.ts`
- 生成 `api.ts`
- 在线预览
- 下载 `types.ts`
- 下载 `api.ts`
- 在接口勾选变化后重新生成

当前实现使用：

- `swagger-typescript-api`

### 4. Mock 数据流程

Mock 数据流程目前固定为 3 步：

1. 接口确认
2. 策略生成
3. 服务状态

#### Step 1：接口确认

支持：

- 按 tag 分组查看接口
- 卡片式选择接口
- `全选`
- `反选`

#### Step 2：策略生成

当前保留的策略项：

- 数据条数

说明：

- 仅对分页接口或 `data` 为数组的接口生效
- AI 模式不再提供开关，代码逻辑中始终开启

#### Step 3：服务状态

每个接口卡片都支持：

- 设置状态码
- 设置延迟毫秒数
- 查看请求调用示例 `curl`
- 查看并编辑 Mock 返回数据
- `热更新`
- `AI 调优`
- 查看请求参数提示
- 标题区域显示操作状态：默认、loading、success、danger

额外支持：

- `刷新 Mock 数据`
  说明：
  会清空当前已选接口在 IndexedDB 中的历史调优数据，重新生成 Mock，并自动热更新到服务
- `重新启动服务`

## AI 调优

项目已经接入右下角 AI 对话弹窗。

当前 AI 调优能力包括：

- 右下角固定聊天入口
- 点击后展示右下角对话面板
- 无 overlay
- 点击弹窗外部不会关闭
- 仅点击关闭按钮关闭
- 支持接口选择器
- 支持多个接口批量调优
- 单接口也可在 Mock 数据预览区域直接调优
- 单接口 AI 调优先点击按钮，再展开输入框填写具体要求
- 单接口 AI 调优提交后，接口标题区域也会同步显示 loading、success、danger 状态

AI 调优逻辑：

- 使用当前接口 Mock 数据作为上下文
- 调用 DeepSeek Chat 返回新的 JSON
- 调优结果立即回填到当前接口 Mock 数据区
- 调优结果写入 IndexedDB
- 后续生成与服务启动时优先读取 IndexedDB 中的调优结果
- AI 提示词限制模型只做最小必要修改
- 当用户只要求调整某个字段时，其他字段保持不变

当前 AI 接口实现位于：

- `app/api/workflow/route.ts`

当前 AI 调用方式：

- SDK：`openai`
- Provider：`DeepSeek`
- Model：`deepseek-chat`
- Key 来源：`DEEPSEEK_API_KEY`

当前支持：

- 服务停止
- 服务重置
- 路由动态配置

Mock 服务相关接口：

- `app/api/mock-server/route.ts`
- `app/api/mock/[...slug]/route.ts`

默认访问前缀：

- `/mock-api`

例如：

```bash

```

## Mock 数据编辑与热更新

服务状态页中的 Mock 数据预览区域已改为可编辑。

当前行为：

- 可以直接编辑 JSON
- 预览区展示的是接口实际返回所使用的 payload 形态
- 点击 `热更新` 后重新配置服务中的该接口返回数据
- 热更新成功后：
  - 接口标题区域显示成功状态
  - 当前接口数据写入 IndexedDB
  - 实际接口返回立即更新
- 热更新失败后：
  - 接口标题区域显示失败状态
  - 页面显示失败提示

刷新 Mock 数据行为：

- 清空当前选中接口在 IndexedDB 中的调优缓存
- 重新生成新的 Mock 数据
- 自动热更新到服务
- 页面显示成功或失败提示

## 接口返回规则

当前接口返回封装规则：

- 默认 `rc` 为 `0`
- 当接口状态码不是 `200` 时，返回 `rc: 1`
- `403` 时返回“请登录”提示

如果 schema 或 mock 数据中存在 `rc` 字段，生成阶段也会固定写成 `0`。

## 图片字段规则

项目内对图片字段做了特殊处理。

识别字段包括但不限于：

- `img`
- `image`
- `ImgSrc`
- `avatar`
- `cover`
- `pic`
- `photo`

规则如下：

- 单图片字段：从 `FIXED_IMAGE_URL` 中随机返回一项
- 图片数组字段：随机返回一个子集
- 当字段本身就是图片数组且长度大于 3 时，会随机截取任意数量返回

相关实现位于：

- `lib/fixed-image-url.ts`
- `lib/mock.ts`
- `app/api/mock/[...slug]/route.ts`

## 数据持久化

当前项目同时使用两类本地存储：

### SessionStorage

用于保存当前工作区基础状态：

- OpenAPI 文本
- 解析结果
- 已选择接口
- 生成后的 TS 内容

### IndexedDB

用于保存 AI 调优后或手动热更新后的接口数据。

特性：

- 以 `workspaceId + endpointId` 为 key
- 同一份 OpenAPI 文档可稳定命中同一个工作区 ID
- 服务启动时优先读取已调优数据
- 刷新 Mock 数据时可按已选接口维度清理历史缓存

相关实现位于：

- `lib/tuned-mock-db.ts`

## 分页接口规则

当前分页处理已经按接口语义做了区分：

### 普通数组接口

如果返回结构是：

```json
{
  "rc": 0,
  "msg": "",
  "data": []
}
```

规则：

- `page/pageSize` 不会影响 `data` 数组数量
- 始终按 Mock 数据预览区中配置的总量原样返回

### 分页对象接口

如果返回结构是：

```json
{
  "rc": 0,
  "msg": "",
  "data": {
    "items": [],
    "total": 0,
    "hasMore": false
  }
}
```

或：

```json
{
  "rc": 0,
  "msg": "",
  "data": {
    "item": [],
    "total": 0,
    "hasMore": false
  }
}
```

规则：

- `page/pageSize` 只影响 `data.items` 或 `data.item` 的返回数量
- `total` 永远取 Mock 数据预览区中配置的数据总量
- 不会因为 `pageSize` 比实际数据量大，就把 `total` 错误放大
- 不传分页参数时，返回预览区中配置的完整内容

## 挑战部分：问题与场景

### 你在工作中遇到了什么困难或挑战？

在前后端联调、测试造数、接口演示这类日常研发场景里，最常见的问题不是“接口没有定义”，而是“接口定义有了，但很难快速得到可联调、可修改、可复用的高质量 Mock 数据”。

具体业务场景：

- 前端开发人员在页面开发阶段，需要尽快接入接口并完成页面联调
- 后端接口尚未完成，或接口已完成但返回数据不稳定
- 测试人员需要覆盖正常、异常、分页、空数据、图片字段等多种返回情况
- 产品或业务同学需要在演示环境中看到更接近真实业务的数据效果

这类问题的典型表现：

- OpenAPI 文档已经存在，但接口多、字段复杂，手工造数据效率很低
- 临时写死 JSON 或本地 mock 文件容易和文档脱节
- 单次改完数据后无法沉淀，下次还要重新改
- 图片、分页、状态码、延迟、异常态等场景覆盖不完整
- 前端想改某个接口返回值时，往往还要找后端或重新改本地代码

本质上，这是“接口定义到可用 Mock 服务”的最后一公里效率问题。

### 过去你是怎么处理的？

过去的传统处理方式通常有几种：

- 手写本地 JSON 文件
- 在代码里临时写死 mock 数据
- 让前端自己维护一套简单 mock
- 等后端接口完成后再联调
- 用通用 mock 工具生成一版数据后，再手工逐个改字段

这些方式的主要痛点：

- 手工成本高
- 接口一多就难维护
- 改完后无法复用
- 数据不够真实
- 多人协作时容易出现“你本地能跑，我这里不一致”
- 一旦接口字段变化，TS 类型、Mock 数据、联调服务要分别更新

### 你希望解决后达到什么效果？（请量化）

希望实现的目标是：

- 单份 OpenAPI 文档从导入到启动可联调 Mock 服务，控制在 3 分钟内
- 单接口返回数据修改时间从 5 到 10 分钟降到 30 秒内
- 接口造数与联调准备时间减少 70% 以上
- 常见联调错误率降低 50% 以上
- 分页、异常态、图片字段等高频场景覆盖率提升到 90% 以上
- AI 调优后可复用，减少重复造数工作 60% 以上

## 解决方案

### 你是如何解决的？（解决方案中应当包含 AI 技术的应用）

我将问题拆成五个连续环节，并把它们整合为一个统一的工作台：

1. 解析 OpenAPI JSON
2. 按接口选择生成 TS 类型
3. 基于 schema 自动生成 Mock 数据
4. 启动本地 Mock 服务并支持热更新
5. 用 AI 对返回数据进行对话式调优，并持久化结果

核心解决思路不是单纯“生成一份 mock”，而是让 OpenAPI 文档直接成为工作区入口，把 TS、Mock、服务、AI 调优、缓存全部串成闭环。

AI 技术的应用方式：

- 在服务状态页和右下角对话面板中接入 AI 调优能力
- 将当前接口已有的 mock JSON 作为上下文
- 通过 `OpenAI SDK + DeepSeek` 调用 `deepseek-chat`
- 让模型只返回合法 JSON
- 将 AI 返回结果直接回填到对应接口的 Mock 数据区
- 同时把调优后的结果写入 IndexedDB，后续优先复用

这样做的价值是：

- 用户不需要改代码，只需要描述“把这个接口改得更像真实电商订单数据”
- 调优后的结果不会丢失
- 服务返回和界面预览始终保持一致

### 关键步骤说明

#### 1. OpenAPI 解析与接口筛选

- 上传 OpenAPI JSON
- 解析接口、请求结构、响应结构、tag、状态码
- 按 tag 分组展示
- 勾选本次需要参与 TS 与 Mock 生成的接口

#### 2. TS 类型生成

- 基于已勾选接口裁剪 OpenAPI 文档
- 使用 `swagger-typescript-api` 生成 `types.ts` 与 `api.ts`
- 支持页面预览和直接下载

#### 3. Mock 数据生成

- 基于响应 schema 递归生成结构化数据
- 针对邮箱、手机号、时间、金额、图片等字段做规则推断
- 对分页接口和数组接口支持批量生成
- 对 `rc` 字段做统一约束，默认固定为 `0`

#### 4. 本地 Mock 服务

- 将当前所有选中接口配置成可访问路由
- 支持状态码、延迟、返回内容热更新
- 支持请求方式匹配与分页数据切片
- 支持图片字段和图片数组字段特殊处理

#### 5. AI 调优与缓存复用

- 在右下角提供对话式 AI 调优入口
- 支持批量选择多个接口统一调整
- 单接口也可以在卡片中直接 AI 调优
- AI 返回结果即时回填
- 写入 IndexedDB，后续服务启动时优先使用历史调优结果
- 点击“刷新 Mock 数据”时，再清理当前接口缓存并重生成

## 新问题与解决

### 在使用新办法的过程中，有遇到新问题吗？

有，主要出现了三类新问题：

1. AI 调优后的数据虽然变了，但服务实际返回没有同步更新
2. 多次生成 Mock 后，历史调优数据和新生成数据之间会互相覆盖
3. 图片字段和图片数组字段在真实业务场景下不够稳定，展示效果不统一

### 如果有，是如何解决的？

针对这些问题，分别做了处理：

- 对“热更新”链路补齐成完整闭环：
  编辑 JSON -> 校验 -> 重配路由 -> 写入 IndexedDB -> 成功/失败反馈
- 为工作区引入 `workspaceId + endpointId` 维度的 IndexedDB 存储：
  让历史调优数据可以稳定命中，也可以按当前接口精确清理
- 增加“刷新 Mock 数据”机制：
  点击后先删除当前选择接口的 IndexedDB 缓存，再重新生成并自动热更新
- 为图片字段增加统一规则：
  从 `FIXED_IMAGE_URL` 中随机取值，数组字段则随机返回任意数量子集
- 为每个接口卡片增加操作状态反馈：
  热更新中显示 Spinner，成功显示 success，失败显示 danger
